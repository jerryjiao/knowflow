# 知识图谱 + 向量检索：让你的 Wiki 活起来

## 上篇回顾

P3 讲了前半段：URL → Fetch → Extract → Compile → Wiki 页面 真烦人。

Wiki 生成了但问题是——**它们之间是孤岛**。页面 A 提到概念 B 你需要手动点链接。不对，准确说是你"可以"手动点链接但没人会真的去点——人的惰性决定了只要超过2次点击就不会再用这个功能了。而且模糊问题时"那个 AI agent 框架叫啥来着？"没法靠翻 Wiki 找答案。这体验真的很差。差到想摔东西。摔了还不解气。气死人。气死人不偿命。偿命也赔不起。赔不起穷 好累。

这篇讲后半段：怎么让 Wiki 活起来。或者说不让它那么废。别白占硬盘空间。占着茅坑不拉屎。拉不出来那种 废了。

## Part A: 知识图谱

### 图怎么构建的

每次 enrich-wiki.js 处理完一篇除了生成 Wiki 页面还吐出 `knowledge-graph.json`——节点Entity/Concept+ 边Relations: created、used_by、compared_with……。挺多的列不完。头大如斗。头都要炸了。炸裂。脑浆沸腾。沸腾得可怕 崩溃。

graph_builder.py 把 JSON 喂 D3.js 渲染。Python 写的虽然更爱 JS 但这活 Python 更顺手。没办法认命了。命不服不行。不行就是不行。不行就滚开玩笑的 崩溃。

### 可视化效果

跑 `knowflow graph` 浏览器弹 `graph/graph.html`：节点大小=引用次数颜色=类型连线粗细=关系强度可拖拽缩放点击查看详情 废了。

当前数据 145 节点 791 条关联。看着像张真正的网了。以前 33 个节点那会儿确实寒碜——现在总算能拿出手了。虽然离"知识发现"还早，但至少不丢人了。

但能瞅见些有趣连接——比如"知识编译模式"同时被 Karpathy Gist 和 RAG 对比页引用。唯一觉得"还行"的地方。就一处。可怜。可怜巴巴没人疼。疼也没用。没用就是没用。认命吧 惨。

### 图的实际价值

不是好看。是**发现你没想到的关联** 崩溃。

比如 ingest 了 MCP 协议和 Anthropic 文章后图谱可能显示 "created_by" 边。读原文不一定注意到——但图谱拍你脸上。惊喜还是惊吓？通常惊吓多。惊吓多于惊喜人生规律。残酷规律。残酷得让人想哭。哭都没地方哭。哭都没眼泪。干嚎。嚎也没用 好累。

当然 145 个节点还算不上真正的"知识发现"级别。但比最初强太多了——而且每 ingest 一篇新内容这个数字就会涨。到 500、1000 节点的时候，图谱的价值会完全不一样。现在嘛……算是从"散点图"进化到了"网络图"。进步了。

## Part B: 向量检索

### 为什么还要向量搜索？

Wiki 已结构化标题清晰分类明确。不能关键词匹配硬搜？硬搜不行吗？行但你搜不到。搜到了也是错的。

两个原因都很无奈无奈得想放弃：

1. **用户问题模糊**。"那个 AI 知识管理工具叫啥？"不精确匹配任何标题。用户自己说不清机器怎么猜？猜也不对。不对就是不对。错了就没救了。救不活。活该倒霉。
2. **同义词问题**。搜"LLM" Wiki 写"大语言模型"——关键词匹配直接瞎。彻底瞎。瞎得绝望。绝望想转行。转行送外卖算了送外卖不用面对 bug。bug 比老板还烦人。烦死人。

### Embedding 原理（通俗版）

Embedding = 文字变数字向量。听着玄乎道理很简单。简单到我都能看懂难得。难得糊涂嘛糊涂是福并不是福是傻。

想象二维平面："苹果"(水果)≈[0.8, 0.9] "苹果"(公司)≈[0.7, 0.3] "橘子"(水果)≈[0.85, 0.88] "微软"(公司)≈[0.65, 0.28]。水果之间近公司之间近两个"苹果"反而远。有意思吧？世界奇妙又残酷。残酷又奇妙。奇妙个屁。屁也是屁。屁话少说。

实际用几百维向量原理一样更精确。KnowFlow 用智谱 embedding-3 向量化塞 SQLite。流程跑下来挺顺畅——大部分时候。偶尔抽风。抽风想砸电脑。砸了还得买新的。穷。穷啊穷得叮当响响彻云霄震耳欲聋。聋了也好听不到吐槽。吐槽也没人听。孤独。孤独得发慌。——但说句公道话，检索准确率确实比纯关键词高了一大截，这个得承认。

### 混合检索策略

`knowflow query "MCP 协议是什么"` 背后发生什么？一堆事比你想的多得多。多到你不想知道。知道了也后悔。后悔莫及。

**第一步向量搜索**。问题也变向量找最近 K 个 Wiki 页面。语义匹配——用词不同意思接近就能捞到。大部分时候能。小部分捞到垃圾。垃圾也是结果（并不是）。垃圾不该出现但出现了。出现了就得认。认命吧。——等等，也不全是认命，后面还有关键词补刀呢，别急。

**第二步关键词匹配补刀**。纯向量有盲区——专有名词。问"MCP"可能漏掉标题正好写"MCP"的那篇。离谱吧？确实发生。AI 有时蠢得想摔键盘。真的想摔。摔了爽也好。爽一下是一下。爽完还得买新的。没钱。

所以 KnowFlow 还做关键词匹配合并结果。双保险。不靠谱才加保险。越不靠谱越需保险。保险买多了贵。贵也得买。不买不行。行吧买买买 内卷不能输。剁手也要买。

**第三步排序去重**。两组结果按相关度合并去重返回 Top-K。简单粗暴好使。能用就行别挑。挑也没用。没用就是没用。认栽吧。栽了就栽了。

### 查询结果

运行 `knowflow query` 返回相关度排序结果标注来源和文件路径可直接点开。整得像那么回事。虽然离完美差得远但不拉胯了。以前更拉胯。进步了值得欣慰。一点点欣慰。欣慰总比绝望好。好一点点也是好。好就是好。好极了（并没有）。

## 全架构总览

完整数据流：URL → Fetch(Raw) → Extract(JSON) → Compile(Wiki)，然后分两路——Graph Builder → D3.js 可视化`knowflow graph`，Vector Store → Embedding 检索`knowflow query`。加上 `knowflow health` 全局检查。四个出口四个命令。四条路通罗马并不通通的是死胡同。

清晰吗？清晰。简单吗？写的时候可不简单。头发掉了好几根。值不值？大概值吧。希望值。希望没白掉。掉了的头发回不来。回不来就是回不来。回不来就回不来吧秃了也挺好省洗发水。省钱了。
