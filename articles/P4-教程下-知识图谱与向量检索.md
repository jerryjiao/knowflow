# 知识图谱 + 向量检索：让你的 Wiki 活起来

## 上篇回顾

P3 讲了前半段：URL → 来源检测 → Fetch → Extract → Compile → Wiki 页面。

Wiki 生成了但问题是——**它们之间是孤岛**。页面 A 提到概念 B 你需要手动点链接。而且模糊问题时"那个 AI agent 框架叫啥来着？"没法靠翻 Wiki 找答案。

这篇讲后半段：怎么让 Wiki 活起来。

## Part A: 知识图谱

### 图怎么构建的

每次 `enrich-wiki.js` 处理完一篇，除了生成 Wiki 页面，还会输出 `knowledge-graph.json`——节点（Entity/Concept）+ 边（Relations: created_by、used_by、compared_with……）。

`graph_builder.py` 读取 Wiki 目录下的所有 `.md` 文件，解析 `[[链接]]` 语法提取实体和关系，然后生成 **vis-network (vis.js) 力导向图**。

### 可视化效果

跑 `knowflow graph` 浏览器弹 `graph/graph.html`：

- **节点大小** = 引用次数
- **节点颜色** = 类型（实体/概念/来源）
- **连线粗细** = 关系强度
- **可拖拽、可缩放、可点击查看详情**

当前数据：**145 个节点、791 条关联**。虽然还在早期，但已经能看到有趣的连接——比如不同文章同时引用同一个概念时，图谱会把它们关联起来。

### 图的实际价值

不是好看。是**发现你没想到的关联**。

比如 ingest 了 MCP 协议和 Anthropic 文章后，图谱可能显示它们之间的 "created_by" 边。读原文不一定注意到——但图谱拍你脸上。

当然 145 个节点还算不上真正的"知识发现"级别。但比最初强太多了——而且每 ingest 一篇新内容这个数字就会涨。到 200+ 节点的时候，图谱的价值会完全不一样。

## Part B: 向量检索

### 为什么还要向量搜索？

Wiki 已结构化标题清晰分类明确。不能关键词匹配硬搜？两个原因：

1. **用户问题模糊**。"那个 AI 知识管理工具叫啥？"不精确匹配任何标题。
2. **同义词问题**。搜"LLM" Wiki 写"大语言模型"——关键词匹配直接瞎。

### Embedding 原理（通俗版）

Embedding = 文字变数字向量。听着玄乎道理很简单。

想象二维平面："苹果"(水果)≈[0.8, 0.9] "苹果"(公司)≈[0.7, 0.3] "橘子"(水果)≈[0.85, 0.88] "微软"(公司)≈[0.65, 0.28]。水果之间近公司之间近两个"苹果"反而远。

实际用几百维向量原理一样更精确。KnowFlow 用**智谱 embedding-3** 将所有 Wiki 页面向量化，存入本地 JSON 索引文件（`.vector-index.json`），查询时实时计算相似度。

### 混合检索策略

`knowflow query "MCP 协议是什么"` 背后发生什么？

**第一步：向量搜索**。问题也变成向量，找最近 K 个 Wiki 页面。语义匹配——用词不同意思接近就能捞到。

**第二步：关键词匹配补刀**。纯向量有盲区——专有名词。问"MCP"可能漏掉标题正好写"MCP"的那篇。所以 KnowFlow 还做关键词匹配合并结果。双保险。

**第三步：排序去重**。两组结果按相关度合并去重返回 Top-K。

### 查询结果

运行 `knowflow query` 返回相关度排序结果，标注来源和文件路径，可直接点开。

## 全架构总览

完整数据流：

```
URL → Fetch(Raw) → Extract(JSON) → Compile(Wiki)
                                    ├─→ graph_builder.py → vis.js 可视化 (knowflow graph)
                                    └─→ vector-store.mjs → Embedding 检索 (knowflow query)
```

加上 `knowflow health` 全局检查。四个命令，四条路。
