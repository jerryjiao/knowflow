# RAG vs Knowledge Compilation

> 📊 综合: 多篇技术文章 + KnowFlow 实践总结

## 概述

RAG（Retrieval-Augmented Generation，检索增强生成）和 Knowledge Compilation（知识编译）是两种不同的 AI 知识处理范式。理解它们的差异对设计知识管理系统至关重要。

## RAG：检索增强生成

### 工作原理
```
用户查询 → 向量检索相关文档 → 将文档注入 Prompt → LLM 生成回答
```

### 核心特点
- **实时检索**: 每次查询都从文档库中搜索
- **无状态**: 不保留上次查询的上下文
- **按需召回**: 只返回与当前查询最相关的片段
- **动态更新**: 文档更新后立即可用

### 优势
- ✅ 信息始终最新（实时索引）
- ✅ 不需要预计算或预处理
- ✅ 支持大规模文档集
- ✅ 实现相对简单

### 劣势
- ❌ 每次查询都消耗大量 token（注入上下文）
- ❌ 检索质量依赖 chunking 和 embedding 质量
- ❌ 难以建立跨文档的知识关联
- ❌ 无法进行深度推理和综合
- ❌ "上下文窗口税"：信息越多，推理能力越弱

## Knowledge Compilation：知识编译

### 工作原理
```
原始素材 → AI 阅读/理解 → 提取关键信息 → 编写/更新 Wiki → 用户查询时直接读取结构化知识
```

### 核心特点
- **预处理**: 素材被"消化"为结构化知识
- **有状态**: 知识随时间积累和演化
- **主动组织**: 建立实体、概念、关系
- **增量维护**: 新素材更新已有知识而非追加

### 优势
- ✅ 查询效率高（直接读取，无需检索）
- ✅ 知识经过提炼，质量更高
- ✅ 支持复杂推理（知识已结构化）
- ✅ token 消耗低（只读相关页面）
- ✅ 可建立知识图谱和网络

### 劣势
- ❌ 需要预处理时间和成本
- ❌ 可能滞后于原始素材更新
- ❌ 编译质量依赖 AI 能力和 prompt 设计
- ❌ 大规模初始编译耗时

## 对比矩阵

| 维度 | RAG | Knowledge Compilation |
|------|-----|---------------------|
| **延迟** | 中等（检索+生成） | 低（直接读取） |
| **Token 成本** | 高（每次注入上下文） | 低（只读结果） |
| **新鲜度** | 实时 | 有延迟（需重新编译） |
| **深度** | 表面（片段级） | 深度（综合多源） |
| **可解释性** | 中等（可追溯来源） | 高（结构化知识） |
| **实现难度** | 较低 | 较高 |
| **适用场景** | FAQ、客服、搜索 | 研究、学习、决策支持 |

## KnowFlow 的混合方案

KnowFlow 采用 **Compilation-first, Retrieval-fallback** 的混合策略：

1. **主路径（Knowledge Compilation）**
   - Ingest 时 AI 编译原始素材为 Wiki 页面
   - 结构化存储在 entities/concepts/sources 目录
   - 建立 wikilink 关系网络

2. **辅助路径（向量检索）**
   - Query 时先搜索 Wiki 页面（轻量 RAG）
   - 找到相关页面后返回完整内容
   - 作为 compilation 的补充

3. **知识图谱**
   - 从 wikilink 自动提取关系
   - 可视化知识网络
   - 发现孤立节点和断裂连接

## 何时选择哪种方案？

### 选 RAG 当：
- 文档频繁更新（新闻、日志、实时数据）
- 用户查询不可预测（开放域问答）
- 快速原型验证
- 文档量极大（百万级）

### 选 Knowledge Compilation 当：
- 需要深度理解和综合
- 知识需要长期维护和积累
- 决策支持和研究场景
- 追求高质量输出
- Token 成本敏感

### 选混合方案当：
- 核心知识稳定但需要补充实时信息
- 需要 deep understanding + broad coverage
- 如 KnowFlow 的使用场景
